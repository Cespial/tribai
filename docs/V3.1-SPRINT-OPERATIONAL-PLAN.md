### ROOT CAUSES PRIORIZADOS

**RC1 (50% de fallos): P@5 penalizado por la formula `relevant_in_top5 / 5`**
- Con 1-2 expected articles y 5 slots, el maximo P@5 alcanzable es 0.20-0.40
- El denominador fijo 5 penaliza queries con 1 solo expected article
- **No es un bug del RAG, es del metrica** — pero aun asi hay mejoras posibles

**RC2 (30% de fallos): Articulos genericos/niche no se diferencian semanticamente**
- Art. 2, 36, 22, 580, 714 tienen contenido generico que se confunde con articulos adyacentes
- El Contextual Retrieval ayudo (+12%) pero no resolvio para articulos de <200 tokens

**RC3 (15% de fallos): IDs compuestos (Art. 260-5, 689-2, 240-1) no matchean en regex del reranker**
- `exactArticlePattern = /Art(?:iculo|\\.)\s*(\d+(?:-\d+)?)/gi` — captura "260-5" pero luego se compara con `id_articulo` que puede ser "Art. 260-5" o "Art. 260 - 5"

**RC4 (10% de fallos): Token budget de 12K favorece articulos ET sobre fuentes externas**
- externalSourceBudgetRatio = 0.40 pero se calcula como `Math.floor(maxTokens * externalRatio)`
- 12,000 * 0.40 = 4,800 tokens para doctrina — insuficiente cuando compite con 10 articulos ET
- Y el budget externo solo se activa cuando hay 3+ chunks con score > 0.70

**RC5 (5% de fallos): Ext Source % = 1.0 siempre es un artefacto de medicion**
- Solo 40 de 300 preguntas tienen `expected_external_sources`
- Para esas 40, el retrieval multi-namespace SI funciona (ext source presence = 1.0)
- Pero la STA (0.512) revela que solo la mitad sobrevive al contexto final

**Archivos:**
- `eval/metrics/retrieval.ts` (linea 3-13):
  ```typescript
  // Nuevo: P@5 adjusted
  export function precisionAtKAdjusted(retrieved, expectedArticles, k) {
    if (expectedArticles.length === 0) return 1; // No expected = perfect (nothing wrong retrieved)
    const topK = retrieved.slice(0, k);
    const relevant = topK.filter(c => expectedArticles.includes(c.metadata.id_articulo));
    return relevant.length / Math.min(k, expectedArticles.length); // Denominator = min(k, expected)
  }
  ```
